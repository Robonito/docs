# Custom Assertions in Robonito

Custom Assertions allow you to validate UI content dynamically using **runtime values** instead of fixed, hardcoded text. 

This is essential when the expected value changes during execution, such as validating:
- Page titles
- Usernames
- Dynamic labels
- API-generated content
- Data entered in previous test steps

Robonito supports variable-based validation using the following syntax:  
`{{!variableName}}`

> **Note:** The `!` explicitly indicates that the value should be resolved dynamically at runtime.

---

## Why Use Custom Assertions?

Traditional assertions verify static text, which often leads to brittle tests:
- **Static:** `"Dashboard"` *(Always the same)*
- **Dynamic:** `"Welcome, Arshad"` *(Changes per user session)*

Custom assertions solve this by validating against stored variables. This ensures your tests continue to pass regardless of the specific data being used in any given session, keeping your test suite robust and flexible.

---

## Variable-Based Assertion Syntax

To reference a dynamically stored variable during a UI assertion, use the double-curly brace notation with an exclamation mark prefix:

**Syntax:**  
`{{!variable_name}}`

### Examples
- `{{!page_title}}`
- `{{!username}}`
- `{{!product_name}}`

During test execution, Robonito automatically replaces these placeholders with the actual runtime value stored in the variable.

---

## Step-by-Step: Using a Custom Assertion

### Step 1: Capture the Dynamic Value
First, capture the target value and save it to a variable using one of the following methods:
- Record a Local Variable during the test execution.
- Capture an Input step value.
- Extract an API variable.
- Capture previous UI content.

**Example:**  
You store the current page title into a variable:  
`page_title = "Local Guide"`

### Step 2: Add the Assertion
1. Open the **Add Assertion** menu in Robonito.
2. Choose **Content Based** → **Custom Expression**.
3. In the **Expected Value** field, enter your formatted variable:  
   `{{!page_title}}`

### Step 3: Execution Behavior
During test execution, Robonito performs the following comparative logic:

| Stage | Value |
| :--- | :--- |
| **Stored Variable** | `Local Guide` |
| **Actual UI Text** | `Local Guide` |
| **Assertion Result** | **✅ PASS** |

---

## Example Workflow

Here is how a custom assertion looks in action:

1. **Navigate:** The user navigates to a page titled `"Local Guide"`.
2. **Save:** You record the variable: `page_title = Local Guide`.
3. **Assert:** You set the expected UI text as `{{!page_title}}`.
4. **Result:** The assertion passes because the text currently displayed in the UI exactly matches the dynamically stored value.

---

## Common Use Cases

### 1. Verify Input Reflection
The user types a dynamic name (e.g., `"Arshad"`) into an input field. You then validate that the profile header matches that exact input: `{{!username}}`.

### 2. Verify Created Item
You create a new product named `"iPhone 15"`. You then verify that the product listing displays that exact product name: `{{!product_name}}`.

### 3. Verify API-Driven Content
An API call returns an order ID (e.g., `"ORD-10231"`). You validate that the success UI screen displays that exact order ID: `{{!order_id}}`.

---

## Comparison: Normal vs. Custom Assertion

| Assertion Type | Example Input | Behavior |
| :--- | :--- | :--- |
| **Static Assertion** | `Local Guide` | Fixed comparison (Hardcoded text) |
| **Custom Assertion** | `{{!page_title}}` | Dynamic comparison (Resolved at runtime) |

---

## Important Rules

Keep the following rules in mind when working with Custom Assertions:

- **Existence:** The variable *must* be defined in a test step **before** the assertion step occurs.
- **Case Sensitivity:** Variable names are strictly case-sensitive.
- **Prefix:** You must use the `!` prefix inside the curly braces (`{{!var}}`) for runtime UI variable comparisons.
- **Versatility:** Custom assertions work seamlessly with both UI-captured variables and API-stored variables.
