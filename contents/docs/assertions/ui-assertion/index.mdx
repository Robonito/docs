# UI / Element-Based Assertions

UI / Element-Based Assertions are used to validate the presence, visibility, count, and styling of specific UI elements on a web page. These assertions ensure that your application's UI components are rendered correctly and behave exactly as expected during test execution.

The **Add Assertion** dialog dynamically updates the options panel when you select **UI / Element-Based** as your assertion type during a recording or while editing a test step.

---

## Choose an Assertion Type

Unlike Content-Based assertions that check for text, UI / Element-Based assertions focus purely on validating structural DOM elements.

Robonito supports the following UI assertion types:

### 1. Element Exists
Asserts that a specific structural element exists physically in the DOM.

- Validates only that the element is present, regardless of whether it is currently visible on the screen or hidden via CSS.
- Useful for ensuring that required base elements are successfully rendered by the browser code.

**Use Case:**  
Verify that a hidden dialog container, core button element, or structural header tag exists on the page.

---

### 2. Element Visible
Asserts that a specific DOM element is actively visible to the user.

- Ensures the element is currently displayed and effectively rendering on the user's screen.
- Crucial for validating UI readiness before subsequent test interactions (like clicking) occur.

**Use Case:**  
Confirm that a critical "Submit" button or warning text banner is actually visible to the user before the test code attempts to click or read it.

---

### 3. Element Count
Validates the exact number of structural elements matching your selected element selector.

**Configuration Options:**
- **Expected Count Value (\*)** – Enter the exact numerical value of expected items.

**Use Case:**  
Verify the total number of items rendered dynamically in a dropdown list, counted table rows, or repeated UI cards in a dashboard grid.

---

### 4. CSS Assertion
Validates the exact output value of a specific CSS stylistic property attached to an element.

**Configuration Options:**
- **CSS Property Name (\*)** – Provide the required CSS key (e.g., `background-color`, `font-size`).
- **Expected Property Value (\*)** – Provide the expected CSS output result (e.g., `#FFFFFF`, `16px`).

**Use Case:**  
Ensure accurate dynamic styling changes in the UI, such as verifying a validation box turns red during an error state or confirming proper typography scales are dynamically applied.

---

## Assertion Configuration

Once a specific UI assertion type is selected, the **Configuration** panel becomes active. 

Depending on your chosen assertion, you can configure matching behaviors:
- **Expected Value (\*)** – The expected value explicitly associated with evaluating the UI element.
- **Match Type** – Define how the evaluation should be handled strictly (e.g., `Exact Match`, `Partial Match`).
- **Case Sensitive** – Enable strictly for enforcing capitalization rules.

These settings define exactly how Robonito evaluates your target UI element during the test's automation execution phase.

---

## Adding the Assertion

After successfully configuring your element assertion logic:

1. **Review:** Double-check your chosen UI assertion type and its active configuration settings.
2. **Add:** Click the **Add Assertion** button on the interface.

The assertion will be closely attached to your selected test step. Robonito evaluates this UI rule exactly at that step and automatically displays the resulting **Pass** or **Fail** timeline status in your test's detailed execution report.
