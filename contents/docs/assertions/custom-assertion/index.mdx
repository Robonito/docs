# Custom Assertions in Robonito

Custom Assertions allow you to validate UI content dynamically using **runtime values** instead of fixed, hardcoded text.

This is essential when the expected value changes during execution, such as:
* Page titles
* User names
* Dynamic labels
* API-generated content
* Data entered in previous steps

Robonito supports variable-based validation using the following syntax:
`{{!variableName}}`

> **Note:** The `!` indicates that the value should be resolved dynamically at runtime.


## Why Custom Assertions?

Traditional assertions verify static text, which often leads to brittle tests:
* **Static:** `"Dashboard"` (Always the same)
* **Dynamic:** `"Welcome, Arshad"` (Changes per user)

Custom assertions solve this by validating against stored variables, ensuring your tests pass regardless of the specific data used in that session.


## Variable Based Assertion Syntax

To reference a variable, use the double-curly brace notation with an exclamation mark:

`{{!variable_name}}`

### Examples
* `{{!page_title}}`
* `{{!username}}`
* `{{!product_name}}`

During execution, Robonito automatically replaces these placeholders with the actual runtime value.


## Step-by-Step: Using Custom Assertion

### Step 1 — Capture Dynamic Value
First, capture a value using one of the following methods:
* Record Local Variable
* Input step value
* Extracted API variable
* Previous UI content

**Example:**
Store the page title into a variable:
`page_title = "Local Guide"`


### Step 2 — Add Assertion
1. Open the **Add Assertion** menu.
2. Choose **Content Based** → **Custom Expression**.
3. In the **Expected Value** field, enter: `{{!page_title}}`


### Step 3 — Execution Behavior
During test execution, Robonito performs the following logic:

| Stage | Value |
| :--- | :--- |
| **Stored Variable** | Local Guide |
| **UI Text** | Local Guide |
| **Assertion Result** | **PASS** |


## Example Workflow

1.  **Navigate:** User navigates to a page titled "Local Guide".
2.  **Save:** Record the variable: `page_title = Local Guide`.
3.  **Assert:** Set the expected text as `{{!page_title}}`.
4.  **Result:** The assertion passes because the UI matches the stored value.


## Common Use Cases

### 1. Verify Input Reflection
The user types a name (e.g., "Arshad"). You then validate the profile header matches: `{{!username}}`.

### 2. Verify Created Item
Create a product named "iPhone 15". You then verify the listing displays: `{{!product_name}}`.

### 3. Verify API Driven Content
An API returns an order ID (e.g., "ORD-10231"). You validate the UI displays: `{{!order_id}}`.


## Comparison: Normal vs. Custom Assertion

| Type | Example | Behavior |
| :--- | :--- | :--- |
| **Static Assertion** | `Local Guide` | Fixed comparison (Hardcoded) |
| **Custom Assertion** | `{{!page_title}}` | Dynamic comparison (Runtime) |


## Important Rules

* **Existence:** The variable must be defined in a step *before* the assertion step.
* **Case Sensitivity:** Variable names are case-sensitive.
* **Prefix:** Use the `!` prefix only for runtime variable comparison.
* **Versatility:** Works with both UI-captured and API-stored variables.
